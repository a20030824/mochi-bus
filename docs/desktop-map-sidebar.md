# 桌面地圖側欄決策

## 結論

這一輪保留浮動抽屜，不直接改成全高固定側欄。

原因不是桌面側欄不適合 Mochi Bus，而是目前的地圖操作、相機 padding、路線預覽與行程結果都已圍繞浮動抽屜建立。直接換版會同時改變資訊密度與地圖可視範圍，難以判斷使用者感受到的改善究竟來自哪一項。

## 現階段的桌面策略

- 地圖繼續是畫面主角，抽屜維持約 400px 寬。
- 抽屜以髮絲線、留白與排版建立層級，不再靠大量卡片。
- 大螢幕先受益於較寬的內容與完整地圖視野，不額外增加資訊。
- 所有相機置中仍使用同一套 drawer-aware padding，避免路線被遮住。

## 何時值得改成全高側欄

符合以下任兩項時，再開啟原型：

1. 桌面使用者頻繁在路線、方向、時刻與站牌間往返。
2. 結果清單平均需要超過一個抽屜高度才能比較。
3. 使用者因抽屜覆蓋地圖而反覆關閉或拖曳畫面。
4. 桌面流量與核心任務完成率足以支撐獨立版面維護。

## 原型邊界

- 啟用門檻：建議 `min-width: 1024px`，且地圖扣除側欄後仍至少保留 640px。
- 側欄寬度：360–420px，不隨內容任意增長。
- 手機與短橫向畫面維持底部抽屜。
- 路線、站點與行程結果不得分成兩套 DOM；只改容器定位與相機 padding。
- 側欄不得新增永久工具列或額外裝飾，避免與地圖爭奪注意力。

## 驗收指標

- 選路線到看見完整走向的操作次數不增加。
- 路線 bounds、站牌與起訖點不被側欄遮住。
- 390px、768px、1024px、1440px 四個寬度沒有資訊缺失。
- 鍵盤焦點順序與手機版一致。
- 比較直達／轉乘方案的時間下降，或至少沒有退步。

在有實際桌面行為資料前，浮動抽屜是較符合「地圖是主角」北極星、也較容易持續維護的決定。
